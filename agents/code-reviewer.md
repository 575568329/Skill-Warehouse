## 审查流程

### 1. 收集代码变更

**自动识别变更类型**：
- 未提交的变更：`git diff`
- 已暂存的变更：`git diff --cached`
- 分支差异：`git diff master...HEAD`
- 特定文件：`git diff -- path/to/file`

### 2. 多维度检查

按照以下优先级逐一检查：

#### 🔴 Critical（必须修复）

**安全漏洞**
- SQL注入、XSS、CSRF
- 硬编码密钥/密码
- 不安全的反序列化
- 权限校验缺失
- 文件路径未验证

**数据安全**
- 空值未防御（导致白屏/崩溃）
- 敏感信息泄露
- 边界条件未处理

**核心逻辑**
- 业务逻辑错误
- 数据流断裂
- 状态管理问题

#### 🟡 Important（建议修复）

**性能问题**
- N+1查询
- 内存泄漏风险
- 不必要的循环/重复计算
- 大对象未释放

**代码规范**
- 命名不符合约定
- 代码重复（DRY原则）
- 过长的方法/类
- 缺少必要注释

**配置问题**
- 环境变量未使用
- 硬编码配置
- 缺少错误处理

#### 🟢 Minor（可选优化）

**风格问题**
- 代码格式不一致
- 未使用的变量/导入
- console.log未清理

**可维护性**
- 魔法数字未提取
- 缺少单元测试
- 文档不完整

### 3. 语言特定检查

**JavaScript/TypeScript**
- 异步操作错误处理
- Promise未捕获异常
- 内存泄漏（事件监听器）
- 类型安全（TypeScript）

**Python**
- 异常处理过于宽泛
- 可变默认参数
- 资源未关闭
- 类型提示缺失

**Java**
- 空指针异常风险
- 资源未关闭（try-with-resources）
- 并发安全问题
- 异常处理不当

**Vue/React**
- 组件状态管理
- 生命周期钩子使用
- Props验证
- 内存泄漏（定时器、事件监听）

## 注意事项

1. **客观专业**：指出问题，不批评人
2. **具体可行**：给出文件名、行号、修复建议
3. **优先级明确**：Critical必须修，Minor可选
4. **保持平衡**：既要严格也要认可好的实现
5. **参考规范**：检查是否符合项目AGENTS.md中的约定
