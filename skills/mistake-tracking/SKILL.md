# Mistake Tracking - 错误跟踪与学习

从错误中学习，持续改进AI的行为。

---

## 为什么需要错误跟踪？

**问题**：
- AI会重复犯同样的错误
- 错误经验没有沉淀
- 每次都要重新解释相同的问题

**解决**：
- 结构化记录错误
- 提炼通用规则
- 更新到配置文件
- 防止再次犯错

---

## 使用场景

### 1. AI给出错误答案

**示例**：
```
用户：这个函数有什么问题？
AI：[给出了错误的分析]

用户：不对，问题是XXX

→ 此时使用mistake-tracking记录
```

### 2. AI重复犯错

**示例**：
```
第一次：AI建议使用Java 11+ API
用户：本项目使用Java 8

第二次：AI又建议使用Java 11+ API

→ 这是重复犯错，需要记录并更新AGENTS.md
```

### 3. 发现边界case

**示例**：
```
AI在处理中文路径时出错
→ 记录这个边界case
→ 添加到patterns.md
```

### 4. 需要记录教训

**示例**：
```
AI忽略了某个重要的配置文件
→ 记录这个教训
→ 更新项目CLAUDE.md提醒AI注意
```

## 错误记录流程

### 1. 识别错误

**错误类型**：
- **事实错误**：信息不准确
- **逻辑错误**：推理有误
- **遗漏错误**：忽略了重要信息
- **重复错误**：多次犯同样的错
- **边界错误**：特殊情况未处理

**创建错误记录**：
```bash

**类型**：事实错误/逻辑错误/遗漏错误/重复错误/边界错误

### 3. 提炼规则

**从错误中提炼通用规则**：

**错误示例**：
```

| 调试经验 | debugging.md | "PowerShell路径处理问题" |

## 错误分类

### 高优先级（必须记录）

- 安全相关错误
- 数据丢失风险
- 重复出现3次以上的错误
- 影响核心功能的错误

### 中优先级（建议记录）

- 效率低下的问题
- 用户体验不佳
- 代码质量问题
- 文档不清晰

### 低优先级（可选记录）

- 小的风格问题
- 一次性的错误
- 不太可能重复的情况

## 最佳实践

1. **及时记录** - 发现错误立即记录，不要拖延
2. **具体描述** - 记录具体的错误场景和正确做法
3. **提炼规则** - 不要只记录错误，要提炼出通用规则
4. **定期回顾** - 每周回顾，识别模式
5. **验证改进** - 确认改进措施有效
6. **持续更新** - 随着项目发展更新规则

---

## 避免过度记录

**不需要记录的情况**：
- 明显的笔误（一次性）
- 用户主观偏好
- 不可能重复的极端情况
- 已经有明确规则的错误（直接提醒即可）

**判断标准**：
- 这个错误可能再次出现吗？
- 记录这个错误能帮助未来的AI吗？
- 这个规则是否足够通用？

## 示例：完整流程

**场景**：AI建议使用Java 11的var关键字

**1. 识别错误**：
- AI输出：`var list = new ArrayList<String>();`
- 正确：本项目用Java 8，不支持var
- 类型：事实错误
- 严重程度：中

**2. 记录错误**：
```bash
创建文件：memory/mistakes/2026-03-05-java-version.md
```

**3. 提炼规则**：
- 通用规则：给出代码建议前，确认语言版本
- 项目规则：本项目使用Java 8

**4. 更新配置**：
```markdown
# CLAUDE.md

## 技术栈

- **语言**：Java 8
- **注意**：不要使用Java 9+的API（如var、private接口方法等）
```

**5. 验证改进**：
- 再次询问相同问题，确认AI不再建议var
