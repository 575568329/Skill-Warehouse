# Planning - 实施计划生成

将设计方案转化为可执行的具体任务列表。

---

## 使用场景

在以下情况下使用此skill：

1. **设计文档已完成** - brainstorming阶段产出了设计方案
2. **需要任务分解** - 将大任务拆分为小任务
3. **项目规划** - 需要清晰的实施路线图
4. **团队协作** - 需要分配任务给不同人员/Agent

---

## Planning流程

### 1. 读取设计文档

**查找设计文档**：
```bash
# 查找最近的设计文档
find docs/plans -name "*design.md" -mtime -7

# 或指定文件
docs/plans/YYYY-MM-DD-<topic>-design.md
```

**提取关键信息**：
- 功能目标
- 技术方案
- 架构设计
- 接口定义
- 依赖关系

### 2. 分解任务

**分解原则**：
- 每个任务 < 1小时（理想状态）
- 任务之间有清晰的边界
- 任务可独立测试验证
- 标注任务之间的依赖关系

**任务分类**：
```
Phase 1: 基础设施
├─ Task 1.1: 创建数据结构
├─ Task 1.2: 配置环境
└─ Task 1.3: 设置测试框架

Phase 2: 核心功能
├─ Task 2.1: 实现核心逻辑
├─ Task 2.2: 实现API接口
└─ Task 2.3: 集成测试

Phase 3: 前端/集成
├─ Task 3.1: UI组件开发
├─ Task 3.2: 前后端联调
└─ Task 3.3: 用户验收测试
```

### 3. 标注依赖关系

**依赖类型**：
- **硬依赖**（Must）：必须先完成前置任务
- **软依赖**（Should）：建议先完成前置任务
- **无依赖**：可并行执行

**依赖图示例**：
```
Task 1.1 (数据结构)
  ↓
Task 2.1 (核心逻辑) ← 硬依赖
  ↓
Task 2.2 (API接口) ← 硬依赖
  ↓
Task 3.2 (前后端联调) ← 硬依赖

Task 1.3 (测试框架) ← 可并行
Task 3.1 (UI组件) ← 可并行
```

### 4. 定义验收标准

**每个任务必须有明确的验收标准**：

```markdown
### Task 2.1: 实现核心逻辑

**描述**：实现XXX功能的核心算法

**输入**：
- 数据A
- 配置B

**输出**：
- 结果C

**验收标准**：
- [ ] 单元测试通过（覆盖率 > 80%）
- [ ] 性能测试达标（响应时间 < 200ms）
- [ ] 代码审查通过
- [ ] 文档更新完成

**预估时间**：2小时

**依赖**：Task 1.1
```

### 5. 风险评估

**识别潜在风险**：
- 技术风险（新技术、复杂算法）
- 依赖风险（第三方库、外部服务）
- 时间风险（预估不准确）
- 资源风险（人力、环境）

**制定缓解措施**：
```markdown
**风险1**：第三方API不稳定
- **影响**：高
- **概率**：中
- **缓解措施**：
  1. 实现降级方案
  2. 添加重试机制
  3. 准备Mock数据用于测试
```

### 6. 生成实施计划

**计划模板**：

```markdown

**预估总时长**：X小时
```

```
## 执行建议

**执行顺序**：
1. 按Phase顺序执行
2. 同一Phase内可并行执行无依赖的任务
3. 每完成一个任务立即测试验证
4. 定期review进度，调整计划

**检查点**：
- 每完成一个Phase进行review
- 发现问题及时调整计划
- 更新实际耗时，优化后续预估
```

## 与其他Skill的协作

### 前置：Brainstorming
```
1. 使用 brainstorming skill → 产出设计文档
2. 使用 planning skill → 产出实施计划
```

### 后续：Execute
```
1. 使用 planning skill → 产出实施计划
2. 使用 coding-agent skill → 按计划执行
3. 使用 code-review skill → 审查代码
```

---

## 最佳实践

1. **渐进式细化** - 先创建高层计划，再逐步细化
2. **留有余地** - 预估时间 × 1.5 = 实际预估
3. **定期更新** - 根据实际进度调整计划
4. **可视化进度** - 使用任务列表跟踪完成情况
5. **及时沟通** - 发现问题立即反馈

---

## 任务粒度建议

**好任务**：
- ✅ "创建UserSchema定义"（30分钟）
- ✅ "实现用户注册API"（1小时）
- ✅ "编写注册功能单元测试"（45分钟）

**不好的任务**：
- ❌ "完成用户模块"（太模糊）
- ❌ "优化性能"（没有具体目标）
- ❌ "修复bug"（缺少描述）
